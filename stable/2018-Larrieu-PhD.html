Artefact

« Analyser la musique avec le code des pièces musicales »
... cela pose problème,

car ce qui est à étudier concerne moins le code que la musique ...

laquelle est engendrée par le code ...

Il faut donc étudier le code au sein d'un système ; un système comprenant l'Homme, le code et la musique.
(Norbert Wiener, Boîte noire )

Code : medium formel pour exprimer des idées.

Vers l'étude des pièces musicales

• Si le code est ciblé : 4X, [tab1~ ], [linedrive~ ] etc.
Il est question des éléments d'une implémentation

Or une implémentation est circonstancielle (choix d'un langage, design ... c'est le reflet d'un programmeur).

• Jupiter, 30 ans, 7 implémentations ...

Faire en sorte d'apporter des connaissances au-delà des implémentations.

Comment ? proposition :

• Concevoir une relation one-to-many entre ce qui est à étudier et les différentes implémentations
(Analogie avec le modèle FRBR)

• Trouver un terme pour dépasser les implémentations

Artefact

→ non « technocentré »
→ vers une étude de l'Homme

→ ensemble d'idées véhiculées par du code
→ le code détermine une apparence de l'artefact
→ l'artefact est abstrait
→ ensemble d'éléments utilisés par une personne pour réaliser une nouvelle implémentation, à partir de la lecture/compréhension d'une autre implémentation

Points faibles et forts de la thèse

Points faibles :

• Plusieurs implémentations pour un même artefact, quelles limites ?

• Module, Paramètre : significations différentes

• Description sans captures d'écran

• Spatialisation non traitée

Points forts :

• Questionner la construction du sens dans l'artefact durant le travail compositionnel

• Artefact et implémentations :
relation one-to-many

• Vers un cadre conceptuel pour décrire les artefacts

• L'artefact se structure en niveaux d'abstraction
